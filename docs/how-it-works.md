# 동작 원리 (How it works)

## 큰 그림 — 두 실행 경로, 한 커맨드

```
[매일 예약 — 스케줄러(작업 스케줄러/cron)]
  run-triage.(ps1|sh)
    → claude -p "/issue-triage headless" --permission-mode dontAsk --model <모델>
       ├─ 이슈 없음   → 로그만 남기고 종료
       ├─ 단순 수정   → 수정 + commit(이슈별) ─┐
       └─ 판단 필요   → [템플릿2] 옵션 코멘트만 (이슈 열어둠)
          (run 끝에) 단순수정 커밋들 ─────────┴→ push 1회 → 각 이슈 [템플릿1] + close

[대화형 세션]
  /session-briefing (세션 첫 행동 — 대기 항목·미push 브리핑)
  /issue-triage      (예약과 완전히 같은 커맨드)
  /resolve-issue N X (판단 대기 이슈를 옵션 X로 구현 → 배치 push → [템플릿3] + close)
```

핵심 설계: **단순한 건 자동으로 끝까지, 판단이 필요한 건 옵션만 정리해 두고 사람이 결정.**
"판단 필요"로 분류되면 봇은 **코드도 커밋도 만들지 않고** 이슈에 옵션 A/B/C(장단점·추천안)만 남깁니다.
사람은 브리핑을 보고 `/resolve-issue 15 B` 한 줄로 결정을 실행합니다.

## 상태는 어디에 사는가 — GitHub 이슈가 SSOT

| 저장소 | 역할 | 신뢰도 |
|---|---|---|
| **GitHub 이슈 코멘트** ([템플릿1/2/3]) | 처리 상태·옵션·결정의 공식 기록 | **SSOT** |
| `git log` (`(#N)` 커밋 메시지) | 코드 변경의 이슈 연결 | 보조 |
| `.claude/automation/log.md` | 로컬 요약 캐시(빠른 훑기용, gitignored) | 캐시 |
| `.claude/automation/decisions/ISSUE-N.md` | 대화형 세션의 옵션 메모(헤드리스는 못 씀) | 캐시 |

이렇게 둔 이유: **헤드리스(dontAsk) 실행은 `.claude/`에 Write할 수 없습니다**(안전 설계 — 아래 참조).
로컬 파일만 믿으면 헤드리스가 처리한 이슈를 놓칩니다. 그래서 모든 판별의 1차 신호는 GitHub 코멘트입니다.

## 멱등성 — 같은 이슈를 두 번 처리하지 않기

매 실행마다 이슈 `#N`에 대해 순서대로 확인:

1. **봇 템플릿 코멘트 존재** (`✅ 자동으로 처리되었습니다` / `🔍 검토가 필요합니다`) → 확정 스킵
2. 이슈가 이미 closed → 확정 스킵
3. `git log --grep="(#N)"` 커밋 존재 → **재구현만 스킵** (커밋이 있는데 이슈가 열려 있고 코멘트가 없다면
   = 직전 run이 push 전에 죽은 것 → 그 이슈를 배치 마감 목록에 넣어 push→코멘트→close로 마무리)
4. 단, 마지막 처리 **후** 사람이 새 코멘트/요구사항을 달았으면 재검토 대상으로 복귀

`(#N)` 괄호 포함 grep이 중요합니다 — `#1`로 검색하면 `#10`, `#15` 커밋에도 걸려 오탐합니다.

## 배치 push — 커밋은 쪼개고 push는 모은다

- **한 이슈 = 한 커밋**: 문제가 생기면 `git revert <해시>` 하나로 그 이슈만 되돌립니다.
- **push는 run/세션당 1회**: 이슈마다 push하면 CI·배포가 이슈 수만큼 돌아갑니다. 커밋 해시가
  이슈를 구분하므로 push를 나눌 이유가 없습니다.
- **🔒 불변식: close·완료 코멘트는 push 성공 후.** 원격에 없는 커밋을 "처리 완료"로 표시하지 않습니다.
  push가 실패하면 커밋은 로컬에 남고 이슈는 열린 채 종료 — 다음 run의 배치 push 또는
  `/session-briefing`의 "미push 커밋" 재개 단계가 실어 나릅니다. **어느 시점에 중단되어도
  거짓 상태가 남지 않는** 크래시-세이프 설계입니다.

## 안전 모델 — 3겹 방어

무인 실행이 코드를 커밋하고 push하는 시스템이므로, 안전은 겹으로 설계되어 있습니다:

```
1층  --permission-mode dontAsk     확인이 필요한 도구는 기본 거부. Write 도구 전면 차단.
2층  settings.json deny/allow      force push·reset --hard·재귀 삭제·원격 변경·curl 등 접두사 차단,
                                   필요한 명령만 allow (git 조회/커밋/push origin, gh issue, 검증 명령).
3층  guard.js (PreToolUse 훅)      명령을 셸처럼 토큰화해 argv 구조 위에서 재검사 —
                                   후치 플래그(git push origin main --force), +refspec,
                                   origin 외 원격/URL push, PowerShell 재귀 삭제까지 차단.
                                   커밋 메시지 "안"의 위험 문자열은 오탐하지 않음.
```

여기에 절차 규칙이 더해집니다: 판단 필요=무커밋, 한 이슈=한 커밋, 검증 실패=판단 필요 강등,
push는 현재 브랜치→origin만, 애매하면 판단 필요.

## 헤드리스의 로그 우회 — `@@LOG@@` 마커

헤드리스는 `.claude/`에 Write할 수 없으므로 log.md를 직접 못 씁니다. 대신 처리한 이슈마다
`@@LOG@@ [이슈] #N: ...` 한 줄을 **stdout**에 출력하고, 권한 모델 밖에 있는 **래퍼 스크립트**가
그 마커를 추출해 log.md에 append합니다. 열린 이슈가 없는 날은 마커를 출력하지 않아
로그가 빈 항목으로 오염되지 않습니다.

## 인증 — 왜 setup-token인가

대화형 `claude login`의 OAuth 액세스 토큰은 대략 하루 안에 만료되고, 갱신은 대화형 세션이 담당합니다.
무인 `-p` 실행은 갱신을 보장하지 못해 다음 날 401로 죽습니다. `claude setup-token`이 발급하는
**장기 토큰**을 gitignored 파일(`oauth-token.secret`)에 두면, 래퍼가 매 실행 시
`CLAUDE_CODE_OAUTH_TOKEN`으로 주입해 로그인 토큰의 수명과 무관하게 동작합니다.
