# /issue-triage — 자동 이슈 트리아지

예약 실행(스케줄러, 헤드리스)과 사람의 수동 실행(대화형)이 **완전히 같은 커맨드**다.
차이는 "판단 필요" 결과의 처리 방식뿐이다(§6).

## ✏️ 프로젝트 설정 (설치 후 이 블록을 프로젝트에 맞게 수정)

- **검증 명령:** 변경 성격에 맞게 실행할 명령 목록. 기본 예시(프로젝트에 맞게 교체):
  - `npm run typecheck` / `npm run lint` / `npm test` / `npm run build`
  - (모노레포·다른 러너면 여기와 `.claude/settings.json`의 allow 목록을 함께 수정한다 — 검증 명령은
    allow에 등록돼 있어야 헤드리스(dontAsk)에서 실행된다.)
- **기본 브랜치:** `main` (다르면 아래 절차의 브랜치명 언급을 치환해 읽는다.)
- **프로젝트 고유 원칙:** 구현 시 반드시 지킬 프로젝트 규칙이 있으면 여기에 적는다(예: 디자인 토큰만 사용,
  카피는 명세와 글자 단위 일치 등). **여기 적힌 원칙을 위반하게 되는 수정은 "단순 수정"이라도 판단 필요로 넘긴다.**

## 0. 실행 모드 판별

- 인자 `$ARGUMENTS` 에 `headless` 가 포함되어 있거나, 지금 세션이 `claude -p`(비대화형·사람 부재)로
  돌고 있으면 → **헤드리스 모드**.
- 그 외(사람이 대화창에서 직접 호출) → **대화형 모드**.

모드에 따라 아래 6단계의 "판단 필요" 처리만 달라진다. 나머지는 동일하다.

> **실행 셸:** git·gh·date 등 명령은 **Bash 도구**로 실행한다(헤드리스 권한이 Bash 기준으로 열려 있음).
> PowerShell 도구로 돌리면 dontAsk 에서 거부되어 실행이 멈출 수 있다.

## 0.5 컨텍스트 예산 — 헤드리스 폭주 방지

**배경:** 헤드리스 `claude -p` 세션은 대화형과 달리 **컨텍스트 자동 압축이 없다** — 읽은 것이
전부 누적되며, 한도를 넘으면 "Prompt is too long"으로 run 전체가 죽는다(원 프로젝트 실사례:
난제 이슈를 헤드리스가 파고들다 22분 뒤 컨텍스트 초과로 종료). 아래 규칙은 헤드리스·대화형
공통이되, 헤드리스에서는 **반드시** 지킨다:

1. **이슈 코멘트는 통째로 읽지 않는다.** 멱등성 ⓪ 체크는 봇 템플릿 헤더 유무만 뽑는다:
   `gh issue view N --json comments --jq '[.comments[].body | select(startswith("## ✅ 자동으로") or startswith("## 🔍 검토가"))] | length'`
   (0이면 미처리, 1+면 처리됨. 재검토 판단이 필요할 때만 마지막 코멘트 1개를 `--jq '.comments[-1].body'`로 본다.)
2. **`log.md`는 전체 Read 금지** — `grep "#N" .claude/automation/log.md | tail -3` 으로 해당 이슈 줄만 본다.
3. **트리아지에 불필요한 대형 문서(프로젝트 인계/이력/변경로그 문서)를 읽지 않는다.**
   트리아지에 필요한 상태는 §1~§3(git/gh)이 전부 준다.
4. **코드 조사는 Grep + 부분 Read(offset/limit)로.** 파일 전체 Read는 300줄 이하 파일만.
5. **검증 출력은 tail만 수집:** `npm test 2>&1 | tail -8` 식으로(전체 출력 금지).
6. **딥다이브 컷(헤드리스):** 원인 규명에 파일 3개 이상을 열어야 하거나 왕복 진단(재현·측정)이 필요해
   보이면 그 이슈는 **즉시 "판단 필요"로 전환**하고 [템플릿 2]에 "원인 진단은 대화형 세션 필요"를 남긴다.
   깊은 디버깅은 헤드리스의 일이 아니다.
7. **log.md 크기 유지(대화형 세션 담당):** log.md가 ~30KB를 넘으면 오래된 항목(최근 2일 제외)을
   `.claude/automation/log-archive.md`로 이전한다(멱등성 1차 신호는 GitHub 코멘트라 아카이브해도 안전).

## 1. 준비

1. 현재 시각을 얻는다 — Bash 도구로 `date "+%Y-%m-%d %H:%M"` (템플릿의 `<처리 시각>`에 사용).
2. 현재 브랜치를 확인한다: `git branch --show-current`. (이 킷은 기본 브랜치 직접 커밋 전략을 가정 — 원격은 `origin` 하나.)
3. `git fetch origin` 로 원격 최신화(선택), 작업 트리가 깨끗한지 `git status` 로 확인한다.

## 2. 열린 이슈 조회

```
gh issue list --state open --limit 50 --json number,title,updatedAt,labels
```
남은 이슈가 없으면 → **로그에 "이슈 없음" 한 줄만 남기고 종료**(§7 로그 규칙, 4단계 이후는 생략).

## 3. 멱등성 체크 — 이미 처리한 이슈 건너뛰기

각 이슈 번호 `#N` 에 대해, 아래 중 **하나라도** 해당하면 기본적으로 **건너뛴다**:
- **(로컬 파일과 독립 — 가장 신뢰할 신호)** 이슈에 봇(자동 트리아지)이 남긴 **템플릿 코멘트가 이미 있다**:
  §0.5-1의 `--jq` 헤더 카운트로 확인한다(코멘트 본문 통째 로드 금지 — 컨텍스트 예산). 카운트 1+ 이면
  **"✅ 자동으로 처리되었습니다"([템플릿1])** 또는 **"🔍 검토가 필요합니다"([템플릿2])** 가 이미 게시된 것 → **건너뛴다.**
  (헤드리스 `claude -p --permission-mode dontAsk` 실행은 Write 도구가 차단돼 `log.md`·`decisions/ISSUE-N.md`
  로컬 캐시를 **못 쓸 수 있다**. 그래서 GitHub 코멘트가 SSOT이자 멱등성의 **1차 신호**다 — 로컬 3개 신호는 보조.)
- `.claude/automation/log.md` 에 `#N` 이 "처리 완료/판단 대기"로 기록되어 있음
- 해당 번호로 커밋이 존재함: `git log --grep="(#N)"` — **반드시 괄호까지 포함**해서 `(#N)` 로 검색한다.
  (`--grep="#N"` 은 부분 문자열 매칭이라 `#1` 이 `#10`·`#15` 커밋에도 걸려 새 이슈를 잘못 스킵한다.
  커밋 메시지는 `<type>: <요약> (#N)` 형식이므로 `(#N)` 이 정확히 일치한다.)
- 결정 문서가 존재함: `.claude/automation/decisions/ISSUE-N.md`

**단, 예외:** 마지막 처리 이후 이슈 내용이 실제로 바뀌었으면 다시 검토 대상에 포함한다.
판단 기준: 이슈 `updatedAt` 이 로그에 기록된 처리 시각보다 이후이거나, 봇이 남긴 마지막
템플릿 코멘트 **뒤에** 사람이 새 코멘트/요구사항을 달았으면 → 재검토(이때는 건너뛰기를 무시하고 4단계로).
`gh issue view N --json comments,body,updatedAt` 로 확인한다.

**배치 미완 재개(중요):** 건너뛰기의 **확정 신호는 ⓪(봇 템플릿 코멘트 존재) 또는 이슈가 이미 closed** 인 경우뿐이다.
②(`git log --grep="(#N)"`)·③(decisions 파일)은 "코드가 이미 있으니 **재구현하지 말라**"는 보조 신호일 뿐 —
이슈가 아직 **열려 있고 봇 코멘트가 없으면**(직전 run 이 배치 push/마감 전에 중단돼 커밋만 남은 경우) 재구현은 건너뛰되
그 이슈를 **§6.5 마감 대기 목록에 포함**해 push·[템플릿1]·close 로 끝낸다(미push 커밋이 붕 뜨지 않게).

## 4. 분류 — 애매하면 항상 "판단 필요"로

**단순 수정 (자동 커밋 → 배치 push+close, §5·§6.5):**
- 오탈자·문구·카피 텍스트 수정
- 색/폰트/여백 등 CSS 값 조정 — **레이아웃 구조 변경은 제외**(그건 판단 필요)
- 기존 설정값 수치 조정(timeout·재시도 횟수 등)
- 원인이 명백한 오타성 버그(변수명 오타, 잘못된 import 경로)
- 기존 패턴을 그대로 복제하는 반복 작업

> 위 ✏️ 「프로젝트 고유 원칙」을 위반하게 되면 "단순 수정"이라도 판단 필요로 넘긴다.

**판단 필요 (코드·커밋 없이 문서만):**
- 새 의존성 추가 / 구현 방식이 여럿이고 트레이드오프 존재
- 데이터 구조·DB 스키마·외부 API 계약 변경
- 요구사항이 모호 / 인증·보안·결제·개인정보 관련
- breaking change / 되돌리기 어렵거나 여러 파일·모듈에 걸친 변경
- 명세·문서에 근거가 없어 **기준을 새로 정해야 하는** 것

경계가 조금이라도 애매하면 → **판단 필요**.

## 5. 단순 수정 처리 (한 이슈 = 한 커밋 — push는 §6.5 배치로 뒤에)

각 단순수정 이슈마다 **커밋까지만** 하고, push·[템플릿1]·close 는 하지 않는다(모아서 §6.5에서 한 번에).

1. 프로젝트 원칙(✏️ 블록)을 지켜 수정한다.
2. 검증: 변경 성격에 맞게 ✏️ 「검증 명령」을 돌린다.
   - **검증이 실패하면 커밋·push 하지 말고**, 이 이슈를 "판단 필요"로 재분류해 6단계(템플릿 2)로 넘긴다
     (자동 push 되는 범위를 안전하게 유지).
3. `git add <바뀐 소스 파일들>`(정확히 지정, `-A` 지양) → `git commit -m "<type>: <요약> (#N)"`.
   **여기서 push·코멘트·close 하지 않는다.**
4. 이 이슈를 **"마감 대기" 목록**에 담아둔다(이슈번호 N·커밋 해시·[템플릿1]에 쓸 변경 요약 1줄을 기억).

## 6. 판단 필요 처리 (코드·커밋 절대 없음)

1. `.claude/automation/decisions/ISSUE-N.md` 를 생성/갱신: 이슈 요약 + 옵션 2~3개(각 장단점) + 추천 옵션과 이유.
   **(대화형 전용 — 헤드리스는 `.claude/` Write 불가라 이 파일을 못 만든다. 헤드리스에선 아래 2단계의 [템플릿 2]
   GitHub 코멘트가 옵션 기록의 SSOT이므로 이 단계를 건너뛰고 바로 2단계로 간다.)**
2. 해당 이슈에 **[템플릿 2]** 코멘트를 게시(이슈는 **열어둔 채** 유지). 코드·git 상태는 건드리지 않는다.
3. 모드 분기:
   - **헤드리스:** 여기서 끝. 사람이 나중에 `/session-briefing` 또는 GitHub 에서 확인.
   - **대화형:** [템플릿 2] 내용을 대화창에도 요약해 보여주고 **"어떤 옵션으로 진행할까요?"** 라고 즉시 질문.
     사람이 답하면 → `/resolve-issue` 와 동일 절차(구현 → commit(배치·push 보류) → 모아서 **push** → [템플릿 3] → close)로 이어간다.
4. **§7 대로 로그 기록** — 대화형은 log.md 에 직접 append, 헤드리스는 `@@LOG@@ [이슈] #N: 판단 필요 → [템플릿2] 게시 (커밋 없음, 이슈 열림)` 를 stdout 에 출력(래퍼가 log.md 에 기록).

## 6.5 배치 마감 — 단순수정 커밋들을 한 번에 push 후 일괄 close

모든 이슈를 §3~§6으로 처리한 뒤 **run 당 한 번만** 수행한다(§5 "마감 대기" 목록이 비어 있지 않을 때).
커밋 해시가 이슈를 구분하므로 이슈마다 따로 push하지 않는다(재배포·CI를 1회로).

1. `git push origin <현재브랜치>` **한 번**으로 이번 run의 단순수정 커밋 전체를 민다.
   (force·다른 브랜치·다른 remote 금지 — 훅/권한이 차단함.)
2. **push 성공을 확인한 뒤에야**, 마감 대기 목록의 **각 이슈**에:
   - **[템플릿 1]** 코멘트를 `gh issue comment N --body-file <임시파일>` 로 게시하고
   - `gh issue close N` 한다.
   > 🔒 **불변식:** [템플릿1]·close 는 **반드시 push 성공 후.** 아직 원격에 없는 커밋을 "처리 완료"로 close하지 않는다.
3. **push 가 실패하면:** 커밋은 로컬에 남기고 이슈는 **열어둔 채** 종료한다(로그에 "push 실패"로 기록).
   미push 커밋은 다음 run 의 배치 push(또는 대화형 `/session-briefing` 재개)가 실어 나른다.
4. **§7 대로 로그 기록** — 마감한 이슈마다: 대화형은 log.md 에 직접 append, 헤드리스는
   `@@LOG@@ [이슈] #N: 단순 수정 → 커밋 <해시> → (배치) push 완료 → close 완료` 를 stdout 에 출력(래퍼가 log.md 에 기록).

## 7. 로그 기록 — 대화형은 직접, 헤드리스는 `@@LOG@@` 마커

공식·상세 기록은 **GitHub 이슈 코멘트**가 원본이고, `log.md` 는 로컬에서 빠르게 훑기 위한 요약이다.

- **대화형**(사람이 실행): `.claude/automation/log.md` **맨 아래에 직접 append**. 실행 방식을 헤더로 구분:
  ```markdown
  ## <YYYY-MM-DD HH:MM> (사용자 수동 실행 - /issue-triage)
  - [이슈] #N: <처리 결과 요약 — 커밋 해시·push·close 또는 판단 대기>
  ```
- **헤드리스**(예약): claude 는 `.claude/`(보호 디렉터리)에 **Write 할 수 없다** — `dontAsk` 가 모든 Write 를
  차단하는 안전 설계 때문이다. 따라서 log.md 에 직접 쓰려 하지 말고, **처리한 이슈마다** 요약을
  **stdout 에 아래 형식의 한 줄로 출력**한다(줄 맨 앞에 마커):
  ```
  @@LOG@@ [이슈] #N: <처리 결과 요약 — 커밋 해시·push·close 또는 판단 대기>
  ```
  래퍼(`run-triage.ps1`/`run-triage.sh`)가 이 `@@LOG@@` 줄들을 뽑아 log.md 에 append 한다(처리 이슈가 있을 때만).
  **열린 이슈가 없으면 마커를 출력하지 않는다**(매일 빈 로그 방지). git/gh 커밋·코멘트·close 는 정상 수행.

**[템플릿 1] 단순 수정 — 자동 처리 완료 (§6.5 배치 push 성공 후, close 직전)**
