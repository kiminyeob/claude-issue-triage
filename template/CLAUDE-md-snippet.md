## 이슈 자동 트리아지 — 운영 모델

> 🚨 **대화형 세션을 열면 — 사용자가 시키지 않아도 가장 먼저 `/session-briefing` 을 실행한다.**
> 최근 처리 이력·미push 커밋·판단 대기 이슈(옵션 포함)를 브리핑하고 "무엇을 진행할지" 물은 뒤 작업을 시작한다.
> (**예외:** 헤드리스 `claude -p` 예약 실행에서는 호출하지 않는다.)

핵심: **단순한 건 자동으로 끝까지, 판단이 필요한 건 옵션만 정리해 두고 사람이 검토 후 수동으로.**

```
[매일 예약 실행 — 스케줄러]
  run-triage.(ps1|sh) → claude -p "/issue-triage headless" --permission-mode dontAsk
     ├─ 이슈 없음   → 로그만 남기고 종료
     ├─ 단순 수정   → 수정 + commit(이슈별) ─┐
     └─ 판단 필요   → 코드/커밋 없이 [템플릿2] 옵션 코멘트만 (이슈 열어둠)
        (run 끝에) 단순수정 커밋들 ──────────┴→ [모아서] push 1회 → 각 이슈 [템플릿1] + close (일괄)

[대화형 세션 — 사람이 필요할 때]
  세션 시작 시 /session-briefing 먼저 실행 (위 🚨 첫 행동 규칙)
  /issue-triage 직접 실행 = 예약과 완전히 같은 커맨드
     ├─ 단순 수정   → 위와 동일(커밋 누적 → 배치 push)
     └─ 판단 필요   → [템플릿2] 게시 + 대화창에 옵션 제시·즉답 → /resolve-issue 로
                      구현 → commit(이슈별·push 보류) → [모아서] push 확인 → 각 이슈 [템플릿3] + close (일괄)
```

**실행 정의(수정·참조는 여기부터):**
- 커맨드(전체 절차·템플릿 전문): `.claude/commands/{issue-triage,session-briefing,resolve-issue}.md`
- 권한 화이트리스트/deny: `.claude/settings.json` · 안전 훅: `.claude/automation/guard.js`
- 스케줄러 래퍼: `.claude/automation/run-triage.ps1`(Windows) / `run-triage.sh`(macOS·Linux)

**분류 — 애매하면 항상 판단 필요:**
- **단순 수정**(자동 commit → 배치 push+close): 오탈자·문구·카피, CSS 값 조정(레이아웃 구조 변경 제외), 기존 설정값 수치 조정, 원인 명백한 오타성 버그, 기존 패턴 그대로 복제. → 프로젝트 고유 원칙(issue-triage.md ✏️ 블록)을 위반하게 되면 단순 수정이라도 판단 필요로 넘긴다.
- **판단 필요**(코드·커밋 없이 옵션 코멘트만): 새 의존성, 구현 방식 여럿·트레이드오프, 데이터 구조·DB 스키마·외부 API 계약 변경, 요구사항 모호, 인증·보안·결제·개인정보, breaking change, 되돌리기 어렵거나 여러 파일·모듈에 걸친 변경, 근거 없는 기준 신설.

**절대 규칙 (자동 세션 안전 경계 — 반드시 지킴):**
- 판단 필요로 분류된 사항은 **코드도 커밋도 만들지 않는다.** [템플릿2] 코멘트만 남기고 이슈는 열어둔다.
- 자동(헤드리스) push는 **오직 "단순 수정" 커밋에 한해**, force 계열 없이 **현재 브랜치를 `origin`으로**만. (settings.json deny + guard.js 훅이 force·다른 브랜치·다른 remote·삭제 push를 원천 차단.)
- **한 이슈 = 한 커밋**으로 작게 쪼갠다 — push된 단순 수정이 문제면 `git revert <해시>` 하나로 되돌린다.
- **push는 배치로 한 번에.** 이슈마다 따로 밀지 않고 처리한 커밋들을 모아 한 번 push한다(재배포·CI 1회, 커밋 해시가 이슈를 구분). commit은 이슈별로 쪼갠 채 유지(revert 단위). **완료 코멘트([템플릿1]/[템플릿3])·close는 push 성공 뒤 일괄** — 아직 원격에 없는 커밋을 완료로 표시하지 않는다. push 실패 시 커밋은 로컬에 남고 이슈는 열린 채, 다음 run/세션이 재개(`/session-briefing` 미push 재개).
- 대화형에서 사람이 이슈 없이 요청한 즉석 수정의 push는 **먼저 확인**한다. 자동 push 예외는 오직 자동 트리아지의 단순 수정 경로뿐.

**멱등성 (중복 처리 방지):** 각 이슈 `#N`에 대해 ⓪이슈에 봇 템플릿 코멘트가 이미 있음(`"✅ 자동으로 처리되었습니다"`/`"🔍 검토가 필요합니다"` 헤더 — **로컬 파일과 독립적인 1차 신호**), ①`log.md` 기록, ②`git log --grep="(#N)"`(괄호 포함), ③`decisions/ISSUE-N.md` 중 **하나라도** 있으면 건너뛴다. 단, 마지막 처리 뒤 사람이 새 코멘트/요구사항을 달았으면 재검토.

**헤드리스 실행 특성:** 헤드리스는 `.claude/`에 **Write 불가**(dontAsk 안전 설계). 그래서 처리한 이슈마다 `@@LOG@@ <한 줄>`을 stdout에 출력 → 래퍼가 `log.md`에 append(열린 이슈 없으면 미출력). `decisions/ISSUE-N.md`는 헤드리스에선 생략 — **GitHub [템플릿2] 코멘트가 옵션 기록의 SSOT.**

**로그:** `.claude/automation/log.md`(로컬·gitignore, 요약 캐시)에 `[이슈]`(트리아지·resolve로 이슈와 연결) / `[직접]`(대화형 즉석 수정)을 구분해 남긴다. **공식·상세 기록은 GitHub 이슈 코멘트가 원본**이고 로컬 파일은 캐시일 뿐.
